# Свой SOCKS5-туннель на VPS для Claude и Google Antigravity

Инструкция «с нуля» для не-администратора. Поднимаем личный SOCKS5-прокси на VPS и заворачиваем в него через **Proxifier** только трафик нужных программ.

```
[Твой ПК] → Proxifier → (интернет) → [VPS за рубежом, SOCKS5] → Claude / Antigravity
```

## 0. Где брать VPS — это решает всё

Прокси отдает трафик **со своего IP**. IP должен быть в стране, где сервис доступен.

| Сервис | Какой IP нужен |
|---|---|
| **Claude** | подходит ЕС (Нидерланды/Германия/Польша) **или** США |
| **Google Antigravity** | часто блок по всему ЕС → нужен именно **США** |

**Вывод: бери один VPS в США — он закроет и Claude, и Antigravity сразу.**
ЕС-вариант ([Timeweb](https://timeweb.cloud/)) оставляй как запасной, если до США не получится оплатить.

### 0.1. Провайдеры с локацией США и оплатой из РФ

| Провайдер | Оплата из РФ | US-локации | Комментарий |
|---|---|---|---|
| **[Aeza](https://aeza.net/)** | МИР / СБП / ЮMoney / крипта (USDT, BTC) | да | проще всего платить из РФ, интерфейс на русском |
| **[PQ.Hosting](https://pq.hosting/)** | крипта + карты | да (несколько городов) | русскоязычная поддержка |
| **[Vultr](https://www.vultr.com/)** | только **крипта** (через BitPay) или реселлер **[Bithost](https://bithost.io/)** (без KYC) | 33 города, много в США | «чистый» западный IP, но платить только криптой |

> Берем минимальный тариф: **1 vCPU / 1 ГБ RAM** — прокси почти не ест ресурсы. ОС везде — **Ubuntu 24.04**.

### А можно платить через PayPal?

Коротко: **с российского аккаунта — нет; с рабочим иностранным — да.** Затык не в провайдерах, а в самом PayPal.

- PayPal свернул работу в РФ в 2022-м, на 2026 без изменений. С **российского** аккаунта нельзя: создать новый, пополнить, отправить или получить деньги — ни внутри страны, ни за границу.
- PayPal годится, **только** если у тебя есть рабочий **иностранный** аккаунт (оформлен на другую страну, пополнен не-российским источником) — или кто-то за рубежом оплатит за тебя.
- Сам PayPal не решает вывод денег из РФ: чтобы пополнить иностранный аккаунт, всё равно нужен не-российский источник средств.

Кто принимает PayPal (нужен рабочий иностранный аккаунт): **VPS** — [Vultr](https://www.vultr.com/), [DigitalOcean](https://www.digitalocean.com/); **резидентные** — [IPRoyal](https://iproyal.com/), [Proxy-Cheap](https://www.proxy-cheap.com/), [Webshare](https://www.webshare.io/), [Decodo](https://decodo.com/) (ex-Smartproxy).

> Практичный вывод для РФ: вместо PayPal используй **крипту (USDT)** — ее принимают те же провайдеры, и она не требует ни карты, ни иностранного аккаунта.

### 0.2. Важная честная оговорка про Antigravity

- Прокси чинит **сетевой/географический** блок (по IP-региону).
- Antigravity дополнительно смотрит на **страну Google-аккаунта**. Если аккаунт оформлен на РФ — одного US-IP может не хватить, аккаунт тоже должен быть в поддерживаемом регионе.
- Google агрессивно режет **дата-центровые IP**. IP любого VPS — дата-центровый. Если US-IP всё равно блокируется — переходи на **резидентный прокси** (см. [Вариант Б](#вариант-б--резидентный-прокси-без-своего-сервера) ниже): домашние IP Google/Anthropic дата-центром не считают.

### Запасной EU-вариант для Claude — Timeweb Cloud

Если оплата США не вышла, для **Claude** хватит ЕС: [Timeweb Cloud](https://timeweb.cloud/) → локация **Нидерланды (Амстердам)**, оплата из РФ работает штатно. Для Antigravity не годится.

### 0.3. Как оплатить криптой пошагово (для новичка)

> 🟢 **Сначала проверь: крипта тебе может быть и не нужна.** [Aeza](https://aeza.net/) и [PQ.Hosting](https://pq.hosting/) принимают **рубли через СБП / МИР** и тоже имеют локацию **США**. Берешь сервер у них — платишь как за обычный онлайн-заказ, без всякой крипты. Крипта обязательна именно для [Vultr](https://www.vultr.com/) (и для тех резидентных прокси, кто не берет рубли).

Если всё же крипта — это **два этапа**: (1) купить USDT за рубли, (2) заплатить ею провайдеру.

#### Этап 1 — купить USDT за рубли

**Вариант A — биржа [Bybit](https://www.bybit.com/), раздел P2P** (курс лучший, комиссия покупателя 0%):
1. Регистрация на [bybit.com](https://www.bybit.com/) + базовая верификация (паспорт).
2. Раздел **P2P** → купить **USDT** → фильтр оплаты **СБП / Сбербанк**.
3. Выбери продавца с высоким рейтингом, укажи сумму — биржа держит USDT в эскроу.
4. Переведи рубли по реквизитам продавца (СБП) → подтверди → USDT придет на аккаунт Bybit. Сеть USDT — **TRC20 (Tron)**, комиссии копеечные.
   - Альтернативы биржи: [BingX](https://bingx.com/), [Bitget](https://www.bitget.com/) — тоже P2P с СБП.

**Вариант B — обменник через [BestChange](https://www.bestchange.ru/)** (без регистрации, в один шаг, на ~1–2 % дороже):
1. На [bestchange.ru](https://www.bestchange.ru/) выбери направление «Сбербанк/СБП → USDT TRC20».
2. Возьми обменник из верха списка (большой резерв, хороший рейтинг).
3. Укажи свой адрес кошелька, переведи рубли → получишь USDT.

> ⚠️ Нужен кошелек для приема. Проще всего — приложение [Trust Wallet](https://trustwallet.com/) (пригодится и на этапе 2). Либо держи USDT прямо на Bybit.
>
> ⚠️ Регуляторика меняется: с июля 2026 в РФ ужесточают контроль P2P-переводов, часть обменников попала под санкции в мае 2026 — бери проверенную биржу/обменник и не гони крупные суммы за раз.

#### Этап 2 — заплатить криптой

**Важно про монету:** Vultr/BitPay принимает BTC, ETH, **LTC**, USDC, BCH, DOGE — но **НЕ USDT**. Поэтому для оплаты держи **LTC (Litecoin)**: одна сеть, копеечная комиссия, быстро. На Bybit USDT → **Convert** в LTC (пара кликов) или сразу купи LTC.

**Путь 1 — напрямую [Vultr](https://www.vultr.com/) через BitPay:**
1. Заведи аккаунт Vultr → **Account → Billing → Make a Payment → Crypto** → укажи сумму (или *Other*).
2. Откроется счет **BitPay**: выбери **Litecoin (LTC)** → покажет адрес, точную сумму и таймер (~15 мин).
3. Плати из **[Trust Wallet](https://trustwallet.com/)**: выведи LTC с Bybit на свой Trust Wallet, затем в кошельке **отсканируй QR** счета BitPay — он сам подставит точную сумму и добавит комиссию сверху. Отправь.
4. Жди зачисления — Vultr кредитует баланс до ~120 минут. Дальше создаешь сервер.

> Почему через кошелек, а не прямо с биржи: BitPay требует **точную** сумму в окно таймера; биржевой вывод режет комиссию из суммы → легко уйти в недоплату (платеж зависнет). Кошелек шлет ровную сумму, комиссию платит сверху.

**Путь 2 — реселлер [Bithost](https://bithost.io/)** (проще, без KYC, без возни с балансом Vultr):
1. На [bithost.io](https://bithost.io/) выбери **Vultr** → план и локацию **США**.
2. Оплати криптой (BTC/LTC) по счету — так же из Trust Wallet по QR.
3. Bithost сам поднимет сервер и выдаст доступ. Чуть дороже, зато без аккаунта Vultr и BitPay-точностей.

#### Что выбрать

| Тебе | Путь |
|---|---|
| Не хочешь крипту вообще | [Aeza](https://aeza.net/) США, оплата **СБП** — самый простой |
| Крипта, но попроще | [Bybit](https://www.bybit.com/) → USDT → Convert в LTC → [Bithost](https://bithost.io/) дает готовый Vultr |
| Крипта, всё сам | [Bybit](https://www.bybit.com/) → LTC → [Trust Wallet](https://trustwallet.com/) → BitPay на [Vultr](https://www.vultr.com/) |

## 1. Создать VPS

> ⚡ **Не делай шаги 3–9 руками — их полностью выполняет скрипт.** Установку и настройку Dante, пользователя, фаервол, fail2ban и проверку автоматизирует [scripts/setup-foreign-vps.sh](scripts/setup-foreign-vps.sh) (одна команда, см. [scripts/README.md](scripts/README.md)). Вручную нужны только шаги 1–2 (создать VPS и подключиться по SSH) — это автоматизировать нельзя. Ручные шаги 3–9 ниже оставлены, чтобы понимать, что происходит, и для отладки.

1. Зарегистрируйся у выбранного провайдера, пополни баланс.
2. Создай сервер: локация **США**, ОС **Ubuntu 24.04**, тариф минимальный.
3. Способ доступа — «по паролю» (проще). Провайдер выдаст **публичный IP** и **пароль root**.

Дальше всё одинаково у любого провайдера. В примерах IP сервера = `203.0.113.45`, порт прокси = `39847` — подставляй свои.

## 2. Подключиться к серверу с Windows

В Windows 10/11 SSH встроен. Открой **PowerShell**:

```powershell
ssh root@203.0.113.45
```

- На вопрос про fingerprint — **сначала сверь** отпечаток с тем, что показывает консоль провайдера (в панели VPS обычно есть «SSH host key fingerprint»). Совпал → `yes`. Это защита от MITM на первом подключении: слепое `yes` доверяет ключу, который мог подсунуть атакующий на канале.
- Введи пароль root (символы при вводе **не видны** — это норма).

Ты внутри, когда видишь приглашение `root@...:~#`.

## 4. Установить Dante (SOCKS5-сервер)

```bash
apt install dante-server -y
```

Узнай имя сетевого интерфейса (понадобится в конфиге):

```bash
ip -o -4 route show to default | awk '{print $5}'
```

Запомни вывод (обычно `eth0` или `ens3`).

## 5. Создать пользователя для авторизации прокси

Пользователь только для логина в прокси — зайти по SSH через него нельзя (`nologin`):

```bash
useradd --system --no-create-home --shell /usr/sbin/nologin proxyuser
passwd proxyuser
```

Придумай **надежный пароль**, введи дважды. Запиши — он пойдет в Proxifier.

---

## 6. Настроить Dante

```bash
nano /etc/danted.conf
```

Удали всё содержимое и вставь это. **Замени два значения:**
- `ens3` → имя интерфейса из шага 4;
- порт `39847` можешь оставить (специально не 1080 — меньше сканируют боты).

```
logoutput: /var/log/danted.log

internal: 0.0.0.0 port = 39847
external: ens3

clientmethod: none
socksmethod: username

user.privileged: root
user.unprivileged: nobody

client pass {
    from: 0.0.0.0/0 to: 0.0.0.0/0
    log: error connect disconnect
}

socks pass {
    from: 0.0.0.0/0 to: 0.0.0.0/0
    log: error connect disconnect
    socksmethod: username
}
```

Сохрани: `Ctrl+O`, `Enter`, `Ctrl+X`.

Запусти и включи автозапуск:

```bash
systemctl restart danted
systemctl enable danted
systemctl status danted
```

Нужен зеленый **active (running)** → `q` для выхода.
Если **failed** → раздел «Если не работает» внизу.

---

## 7. Открыть порт в фаерволе

```bash
ufw allow OpenSSH
ufw allow 39847/tcp
ufw enable
```

Подтверждение → `y`.

> ⚠️ Сначала разреши **OpenSSH**, потом включай ufw — иначе отрежешь себе SSH-доступ.

Проверь панель провайдера на наличие **облачного фаервола** (cloud firewall): если включен — добавь там TCP-порт `39847`.

---

## 8. Защита от перебора пароля

```bash
apt install fail2ban -y
systemctl enable --now fail2ban
```

Нестандартный порт + надежный пароль + fail2ban — достаточно для личного использования.

## 9. Проверить прокси прямо с сервера

```bash
curl --socks5 proxyuser:ТВОЙ_ПАРОЛЬ@127.0.0.1:39847 https://api.ipify.org
```

Должен вывести **IP твоего VPS** (американский). Значит SOCKS5 работает.
Отключиться от сервера: `exit`.

## 10. Настроить Proxifier (Windows)

> ⚡ Можно не настраивать прокси-цепочку в Proxifier вручную по шагам ниже: на Windows-ПК запустите [scripts/make-proxifier-profile.ps1](scripts/make-proxifier-profile.ps1) (например, `.\make-proxifier-profile.ps1 -ProxyHost <IP_РФ_сервера>`) — он соберет готовый к импорту профиль `.ppx` с SOCKS5-прокси `IP:1080`, логином `relayuser` (пароль в файл не пишется — введете в GUI один раз), правилом «AI» для нужных приложений и резолвом DNS через прокси; затем в Proxifier выберите `Profile → Import Profile…`. Если ваша версия Proxifier откажется импортировать файл (`Cannot import / Unsupported profile`) — настройте прокси вручную по этому разделу теми же значениями `Host/Port/User`, а профиль используйте как шпаргалку.

Сайт: [proxifier.com](https://www.proxifier.com/)

### 10.1. Добавить прокси
**Profile → Proxy Servers → Add:**
- **Address:** `203.0.113.45` (IP VPS)
- **Port:** `39847`
- **Protocol:** `SOCKS Version 5`
- **Authentication:** ☑ → **Username** `proxyuser`, **Password** твой пароль
- Нажми **Check** — должно пройти успешно.

### 10.2. Защита от DNS-утечки (важно для обхода блокировок по доменам)
**Profile → Name Resolution** → включи **«Resolve hostnames through proxy»**.
Тогда DNS-запросы тоже идут через VPS.

### 10.3. Правила — заворачиваем ТОЛЬКО нужные программы
**Profile → Proxification Rules:**
- Правило **Default** оставь **Direct** (обычный трафик идет напрямую, на полной скорости).
- **Add** новое правило:
  - **Name:** `AI`
  - **Applications:** имена `.exe` через `;`, например:
    - Google Antigravity → `Antigravity.exe`
    - Claude Desktop → `Claude.exe`
    - доступ к `claude.ai` через браузер → добавь браузер (`chrome.exe`); лучше — отдельный портативный браузер только под это, чтобы не гнать через VPS весь веб
  - **Action:** выбери созданный SOCKS5-прокси.

Точное имя `.exe`: открой программу → **Диспетчер задач** → правый клик по процессу → **Подробно** → имя файла.

После этого только Claude/Antigravity ходят через США, остальной интернет — напрямую.

## Chrome через личный VPS

> Уточнение: **MTProto-прокси — только для Telegram.** Для Chrome нужен твой **SOCKS5**: РФ-релей (`IP:1080`, `relayuser`) или напрямую зарубежный Dante (`IP:39847`, `proxyuser`).

**Главный подвох:** Chrome (Chromium) **не умеет авторизацию в SOCKS5** (логин/пароль). Поэтому нацелить Chrome прямо на релей с паролем флагом `--proxy-server` не выйдет — нужен один из обходов ниже.

### Способ 1 — через Proxifier (если уже настроен), проще всего

Ничего в самом Chrome не трогаешь. В [Proxifier](https://www.proxifier.com/) добавь `chrome.exe` в правило на твой SOCKS5 (Proxifier сам выполняет авторизацию). См. раздел 10. Подходит, если Proxifier — твой хаб.

### Способ 2 — локальный SSH-туннель + отдельный Chrome (без Proxifier)

SSH-туннель дает **локальный SOCKS5 без пароля** на `127.0.0.1:1080` — а с localhost проблема авторизации отпадает.

1. Подними туннель к зарубежному VPS: [scripts/proxy-tunnel.ps1](scripts/proxy-tunnel.ps1) `-VpsIp <IP_зарубежного_VPS>` (выход будет с иностранного IP).
2. Создай ярлык Chrome для отдельного, проксированного профиля. Target ярлыка:
   ```
   "C:\Program Files\Google\Chrome\Application\chrome.exe" --proxy-server="socks5://127.0.0.1:1080" --user-data-dir="%USERPROFILE%\chrome-vps"
   ```
   `--user-data-dir` — отдельный профиль, чтобы **основной** Chrome ходил напрямую. Схема `socks5://` гонит через прокси и DNS (нет утечки/обхода по DNS). Запускаешь этот ярлык — весь его трафик идет через VPS.

### Способ 3 — расширение SwitchyOmega (точечно по сайтам, с переключателем)

Если хочешь гнать через VPS **только** `claude.ai`, а остальное напрямую, и быстро вкл/выкл:

1. Подними локальный SSH-туннель (Способ 2 — без пароля на `127.0.0.1:1080`).
2. Поставь расширение **Proxy SwitchyOmega** → создай профиль **SOCKS5** `127.0.0.1:1080`.
3. Настрой **auto switch** по доменам: `claude.ai`, `*.anthropic.com` → через прокси; всё прочее → Direct.

SwitchyOmega + локальный туннель обходит ограничение Chromium на SOCKS-авторизацию, т.к. localhost-прокси без пароля.

### Чего НЕ делать

- Не вписывай релей с паролем (`1080 relayuser`) прямо в Chrome флагом — Chrome не отправит SOCKS5-пароль, соединение не встанет. Авторизацию умеет Proxifier (Способ 1), либо localhost-туннель снимает ее необходимость (Способы 2–3).
- Не меняй **системный прокси Windows** ради Chrome — это завернет вообще весь трафик системы; лучше отдельный профиль/ярлык или Proxifier.
- Antigravity — это **десктоп-приложение**, а не сайт: через Chrome его не проксировать, для него — Proxifier (раздел 10) или резидентный US-IP (Вариант Б).

## Альтернатива: SSH-туннель без установки софта на сервер (безопаснее)

> ⚡ SSH-туннель и его автоперезапуск можно не держать руками: на Windows-ПК запустите [scripts/proxy-tunnel.ps1](scripts/proxy-tunnel.ps1) (например, `.\proxy-tunnel.ps1 -VpsIp <IP_зарубежного_VPS> -VpsUser proxyuser -LocalPort 1080`) — он поднимет локальный SOCKS5 на `127.0.0.1:1080` через прямой `ssh -D` к зарубежному VPS и сам будет держать его живым (авто-переподключение с нарастающей паузой при сбоях). Первый запуск сделайте вручную в консоли, чтобы принять host key и/или ввести пароль; затем для автозапуска при входе в систему — `.\proxy-tunnel.ps1 -VpsIp <IP> -Install` (регистрирует Scheduled Task), удаление — `-Uninstall`.

Ничего не ставим на сервер — SSH сам поднимает локальный SOCKS5. В PowerShell запусти и **не закрывай** окно:

```powershell
ssh -D 1080 -N -C root@203.0.113.45
```

SOCKS5 появится на `127.0.0.1:1080`. В Proxifier тогда укажи:
- **Address:** `127.0.0.1`, **Port:** `1080`, **SOCKS5**, **без авторизации**.

**Плюс:** наружу торчит только SSH (порт 22), нечего сканировать.
**Минус:** надо держать окно открытым; при обрыве — перезапускать (для автозапуска есть `plink` из PuTTY или Bitvise).
При этом варианте шаги 4–9 **не нужны**.

### Для Claude Code — способ проще: запускать его ПРЯМО на VPS (без Proxifier)

1. Арендуй VPS вне РФ (для Claude хватит **ЕС**, напр. Амстердам; США не обязателен), ~2 ядра / 4 ГБ.

> Это решает именно **Claude**. Для **Antigravity** способ не годится (это GUI-IDE, а не CLI) — там остается туннель + по возможности резидентный US-IP.

### Dante тяжеловат? На Habr чаще советуют 3proxy

Для своего SOCKS5 новичкам на Habr рекомендуют **[3proxy](https://habr.com/ru/articles/742374/)** — легче Dante, проще конфиг. А самый ленивый путь — **[SOCKS5 поверх SSH](https://habr.com/ru/articles/974040/)** (наш `ssh -D` выше), его там так и зовут «пока чайник закипает».

### ⚠️ Нюанс DPI / РКН (июнь 2026)

Различай два разных барьера:
- **Блок на стороне сервиса** (Claude/Antigravity отказывают российскому/дата-центровому IP) → решается локацией VPS и, для Antigravity, резидентным IP.
- **Блок на стороне РКН** (ТСПУ режет сам канал до сервера) → в июне 2026 ТСПУ активно ловит **VLESS + Reality** (по IP/AS, fingerprint uTLS, числу параллельных TLS). Но **простой SSH-туннель и SOCKS5 на случайном порту обычно проходят** (SSH — критичная инфраструктура), поэтому для нашей задачи их хватает.

**Если канал до VPS начнут резать**, эскалация (всё есть на Habr):
- поднять **VLESS + Reality** через панель **[3X-UI](https://habr.com/ru/sandbox/285968/)**;
- или кнопочно через **[Amnezia VPN](https://habr.com/ru/companies/amnezia/articles/832610/)** (сама настроит сервер);
- или сменить протокол на **Shadowsocks-2022 / WireGuard / xHTTP** ([обзор протоколов](https://habr.com/ru/articles/799751/)).

Эти протоколы — про обход РКН, а не про геоблок сервиса; для Claude/Antigravity они нужны, только если простой туннель перестанет ходить.

## А если есть свой сервер в РФ?

Короткий вердикт: **РФ-сервер не может быть выходным узлом и не обходит DPI сам по себе.** Его IP — российский, а значит [Antigravity](https://antigravity.google) и часть сервисов его геоблокируют ровно так же, как ваш домашний IP. И DPI он не обходит: трафик с РФ-сервера за границу уходит через тот же магистральный ТСПУ, что и у всех. Поэтому РФ-сервер полезен только как **всегда-онлайн relay/хаб** перед зарубежным VPS: он держит постоянное соединение, к нему удобно цепляться с любого устройства по логину/паролю (домашний IP может плавать — это не мешает), а наружу всё всё равно выходит через зарубежный выходной узел.

### Схема

```
[ПК / телефон]
      |  SOCKS5 (логин/пароль relayuser)
      v
[твой РФ-сервер :1080]  <-- 3proxy, своя авторизация
      |  parent SOCKS5 (логин/пароль proxyuser)
      v
[зарубежный VPS :39847] <-- Dante, выход с иностранного IP
      |
      v
   Claude / интернет
```

### Пошагово (порядок важен)

> ⚠️ **Перед запуском.** Скрипты лежат в [scripts/](scripts/). Все `setup-*.sh` подключают общую библиотеку [scripts/_lib.sh](scripts/_lib.sh) — **копируй ее в тот же каталог, что и скрипт, и запускай файлом** (`sudo bash имя.sh`), а **не через pipe** (`ssh ... 'bash -s' < ...` не найдет `_lib.sh`). Залей на сервер `_lib.sh` + нужный скрипт (`scp` или вставь через `nano`). Если правил в **Windows-редакторе** (CRLF) — сначала убери возвраты каретки во **всех** `.sh` (включая `_lib.sh`), иначе bash ругнется `$'\r': command not found`:
>
> ```sh
> sed -i 's/\r$//' *.sh
> ```

**Шаг 1. Поднять выходной узел на зарубежном VPS.**
На зарубежном VPS от root запустите [scripts/setup-foreign-vps.sh](scripts/setup-foreign-vps.sh), обязательно указав `ALLOW_FROM` = публичный IP вашего РФ-сервера (иначе SOCKS-порт не откроется в ufw — это безопасное поведение по умолчанию):

```sh
ALLOW_FROM=IP_РФ_СЕРВЕРА bash setup-foreign-vps.sh
```

Из итогового блока **запишите 4 значения**: IP сервера, порт SOCKS (`39847` по умолчанию), логин (`proxyuser`) и пароль (если не задавали свой — он сгенерирован автоматически, сохраните его).

**Шаг 2. Вписать эти данные на РФ-сервере и запустить релей.**
Откройте [scripts/setup-ru-relay.sh](scripts/setup-ru-relay.sh) и в блоке КОНФИГУРАЦИИ подставьте значения из шага 1:

| Переменная в setup-ru-relay.sh | Что подставить (из шага 1) |
|---|---|
| `FOREIGN_VPS_IP` | IP зарубежного VPS |
| `FOREIGN_SOCKS_PORT` | порт SOCKS (`39847`) |
| `FOREIGN_USER` | логин Dante (`proxyuser`) |
| `FOREIGN_PASS` | пароль Dante |

Остальное можно оставить по умолчанию: локальный листенер `LOCAL_PORT=1080`, логин клиента `LOCAL_USER=relayuser`, пароль `LOCAL_PASS` оставьте пустым — он сгенерируется через openssl и появится в итоговой сводке. Запустите:

```sh
sudo bash setup-ru-relay.sh
```

Скрипт сам проверит цепочку (`curl` через локальный SOCKS5) — в конце он покажет иностранный IP, если всё связалось. Запишите `LOCAL_PASS` из блока ИТОГОВАЯ СВОДКА.

**Шаг 3. Указать РФ-сервер в Proxifier (Windows).**
Добавьте SOCKS5-прокси:
- Хост = публичный IP вашего **РФ-сервера**
- Порт = `1080` (`LOCAL_PORT`)
- Authentication = `relayuser` / ваш `LOCAL_PASS`

Проверка с любой машины:
```sh
curl --socks5 relayuser:ПАРОЛЬ@IP_РФ_СЕРВЕРА:1080 https://api.ipify.org
```
Должен вернуться иностранный IP зарубежного VPS.

### Минимальная альтернатива без 3proxy: autossh одной командой

Если не хотите ставить 3proxy, на РФ-сервере можно поднять туннель прямо в SSH к зарубежному VPS — SOCKS5 откроется локально на `1080`, а трафик пойдет через VPS:

```sh
apt install autossh
autossh -M 0 -f -N -D 0.0.0.0:1080 root@FOREIGN_VPS_IP
```

**Предупреждение:** у такого SOCKS **нет пароля** — кто угодно, кто достучится до `1080` на РФ-сервере, получит ваш зарубежный выход. Обязательно закройте порт фаерволом только на свой домашний IP:

```sh
ufw allow from ВАШ_ДОМАШНИЙ_IP to any port 1080 proto tcp
ufw deny 1080/tcp
```

Поэтому autossh годится при статичном домашнем IP; при плавающем IP удобнее именно [scripts/setup-ru-relay.sh](scripts/setup-ru-relay.sh) (там авторизация по паролю, а не по доверенному IP).

### Бонус: полноценный VPN для телефона и всех устройств

Если нужен не точечный прокси (Proxifier работает только на ПК и только для указанных приложений), а **весь** трафик устройства — поднимите WireGuard на зарубежном VPS скриптом [scripts/setup-foreign-wireguard.sh](scripts/setup-foreign-wireguard.sh). Он выдает готовый клиентский конфиг и QR-код: телефон сканирует QR — и весь его трафик (включая мессенджеры, мобильные приложения и системные службы, которые не умеют в SOCKS) выходит с иностранного IP. WireGuard удобнее SOCKS, когда нужен телефон, приложения без поддержки прокси или просто «включил и забыл» на всё устройство. Клиент при этом может цепляться к VPS напрямую — РФ-релей для WireGuard не обязателен.

### Бастион: закройте SSH зарубежного VPS на IP РФ-сервера

Чтобы к зарубежному VPS можно было подключаться по SSH **только** с публичного IP вашего РФ-сервера (бастиона), не настраивайте `ufw` вручную — используйте два скрипта **строго в этом порядке**, иначе рискуете закрыть себе доступ:

1. На **РФ-сервере** запустите [scripts/setup-ru-sshkey.sh](scripts/setup-ru-sshkey.sh) (в интерактивной сессии — потребуется один раз ввести пароль VPS). Он создает ed25519-ключ, копирует его на VPS, проверяет беспарольный вход и печатает **публичный egress-IP РФ-сервера**. Запишите этот IP и убедитесь, что строка `OK: вход по ключу … работает без пароля` появилась.
2. **Только после того, как вход по ключу с РФ-сервера на VPS реально работает**, на **зарубежном VPS** запустите [scripts/setup-foreign-bastion.sh](scripts/setup-foreign-bastion.sh), передав egress-IP РФ-сервера:
   ```sh
   RU_SERVER_IP=<egress-IP_РФ-сервера> bash setup-foreign-bastion.sh
   ```
   Скрипт сначала добавит узкие правила `allow from <IP>` ко всем реальным SSH-портам, затем спросит подтверждение и лишь потом удалит широкие правила (включая парные IPv6). Правило SOCKS-порта (`39847`) он не трогает. Для неинтерактивного запуска добавьте `CONFIRM=yes`.

> ⚠️ **Риск локаута.** Запускайте бастион **на зарубежном VPS** (у него обычно есть аварийная консоль провайдера), а не на РФ-сервере. **Перед** запуском откройте консоль провайдера (KVM/serial/VNC/recovery) и в отдельной сессии убедитесь, что SSH к VPS проходит именно с РФ-сервера. При неверном/изменившемся `RU_SERVER_IP` (опечатка, динамический/CGNAT-адрес) вы заблокируете себе SSH — спасет только консоль провайдера.

Ручной вариант как fallback (если хотите запереть SSH без скрипта): на VPS выполните `ufw allow from <egress-IP_РФ-сервера> to any port 22 proto tcp`, проверьте вход с РФ-сервера и затем `ufw delete allow OpenSSH`. Откат при потере доступа — только через консоль провайдера: `ufw allow OpenSSH`.

### Чего РФ-сервер НЕ дает

- **Не снимает геоблок.** Его IP российский — для сервисов, которые блокируют РФ, выходным узлом он быть не может.
- **Не обходит DPI.** Его исходящий трафик за границу идет через тот же ТСПУ; фильтрация применяется к нему так же, как к вашему домашнему каналу. Обход обеспечивает только зарубежный узел (Dante/WireGuard) на той стороне.
- **Добавляет +1 хоп задержки.** Цепочка `ПК -> РФ -> зарубеж` всегда медленнее прямого `ПК -> зарубеж`. РФ-релей оправдан удобством (общий хаб, авторизация по паролю), а не скоростью.
- **Antigravity он не спасает.** Antigravity нужен резидентный US-IP; ни РФ-релей, ни обычный VPS-IP эту задачу не решают — для нее нужен другой класс выходного IP.

## Вариант Б — резидентный прокси (без своего сервера)

Резидентный прокси — это **готовый** сервис: ты ничего не настраиваешь и не арендуешь VPS. Покупаешь доступ к пулу «домашних» IP, получаешь параметры шлюза (`host:port` + логин/пароль) и сразу вставляешь их в Proxifier (раздел 10). Шаги 1–9 **полностью заменяются** этим разделом.

### Зачем это нужно именно для Antigravity

Anthropic и особенно Google маркируют IP дата-центров (а IP любого VPS — дата-центровый) и могут резать. Резидентные IP принадлежат реальным домашним провайдерам — для сервиса это «обычный пользователь», блокируют их гораздо реже. Это главный (часто единственный) способ, когда VPS не пускают.

### Свой VPS vs резидентный прокси

| | Свой VPS + Dante | Резидентный прокси |
|---|---|---|
| Настройка сервера | да (шаги 1–9) | **нет**, только Proxifier |
| Тип IP | дата-центр (могут резать) | домашний (режут редко) |
| Цена | фикс ~$3–6/мес, трафик не лимитен | per-GB или per-IP (см. ниже) |
| Скорость | высокая | ниже (но для чата/кодинга ок) |
| Для Claude (EU/US) | ✅ дешево и достаточно | ✅ избыточно |
| Для Antigravity | ⚠️ может блокировать | ✅ **то, что нужно** |

**Рекомендация:** для Claude хватит своего VPS (дешевле). Для Antigravity — резидентный, и лучше **статический резидентный (ISP)**, см. ниже.

### Два типа резидентных прокси — бери правильный

| Тип | Оплата | Поведение IP | Для нас |
|---|---|---|---|
| **Rotating residential** | per-GB (~$1–8/ГБ) | IP меняется каждый запрос или по таймеру | нужны **sticky-сессии**, иначе логин рвется; агентный Antigravity ест трафик → дорого |
| **Static residential / ISP** | per-IP (~$2.5–10/IP в мес), трафик обычно безлимит | **один фиксированный** домашний IP | ✅ **лучший выбор**: стабильный IP для долгих сессий, нет счета за гигабайты |

> Для Claude/Antigravity бери **Static residential (ISP), локация США**. Один фиксированный IP, безлимитный трафик, сессии не рвутся — и платишь предсказуемо.

### Провайдеры с оплатой из РФ (крипта)

Корпоративные ([Bright Data](https://brightdata.com/), [Oxylabs](https://oxylabs.io/)) требуют бизнес-KYC и часто не продают в РФ — **не бери**. Подходят самообслуживаемые с криптой:

| **[Proxy-Cheap](https://www.proxy-cheap.com/)** | крипта (BTC/USDT/ETH/LTC) + карта | rotating **и** static residential, SOCKS5 | бюджетно, есть отдельные страницы оплаты криптой |

### Настройка в Proxifier

В личном кабинете провайдера найди **endpoint / proxy generator**: выбери протокол **SOCKS5**, страну **США**, тип — **Static (ISP)** или, если rotating, включи **Sticky session**. Получишь строку вида `host:port:username:password`. Перенеси ее в Proxifier:

**Profile → Proxy Servers → Add:**
- **Address:** шлюз провайдера (например `geo.iproyal.com`)
- **Port:** из кабинета (например `12321`)
- **Protocol:** `SOCKS Version 5`
- **Authentication:** ☑ → **Username** и **Password** из кабинета

> ⚠️ **Sticky-сессия (только для rotating):** у rotating-планов «постоянство» IP кодируется **внутри username**, например `user-cc-us-session-ab12cd-sessTime-30m`. Синтаксис у каждого провайдера свой — **копируй username из их генератора как есть**, не сокращай. Иначе IP сменится посреди логина и сессия слетит. У **static (ISP)** этого не нужно — IP всегда один.

Дальше — всё как в основной инструкции:
- раздел **10.2** — включи **«Resolve hostnames through proxy»** (защита от DNS-утечки);
- раздел **10.3** — правило только на `Antigravity.exe` / `Claude.exe`, **Default = Direct**.

### Про деньги — честно

- **Static (ISP)** — фиксированная плата за IP в месяц, трафик безлимит → предсказуемо, бери этот.
- **Rotating (per-GB)** — Antigravity как агентный IDE может прокачивать много данных; на $1–8/ГБ счет растет незаметно. Если берешь rotating — следи за расходом трафика в кабинете.

### Как подключить residential к этой системе: способ A или B

У провайдера берешь шлюз `host:port:login:pass` (SOCKS5, **static/ISP**, локация США). Дальше — два способа, оба уже «прикручены»:

| | **A. Proxifier → прямо на residential** | **B. Residential как `parent` РФ-релея** |
|---|---|---|
| Цепочка | `[ПК] → residential-шлюз → интернет` | `[ПК] → РФ-релей (3proxy) → residential-шлюз → интернет` |
| Свой VPS нужен? | **Нет** | Да (РФ-релей; зарубежный Dante не нужен) |
| Кто видит твой домашний РФ-IP | residential-провайдер | только РФ-релей (провайдер видит IP релея) |
| Как включить | сгенерь профиль на шлюз: `make-proxifier-profile.ps1 -ProxyHost <шлюз> -ProxyPort <порт> -ProxyUser <логин>` ([scripts/make-proxifier-profile.ps1](scripts/make-proxifier-profile.ps1)) | впиши `RESI_HOST/RESI_PORT/RESI_USER/RESI_PASS` (и `RESI_TYPE=socks5`/`http`) в [scripts/setup-ru-relay.sh](scripts/setup-ru-relay.sh) и запусти — релей сам направит выход на residential |
| Когда выбирать | один ПК; не нужен общий хаб/несколько устройств; шлюз достижим из РФ напрямую | уже держишь РФ-релей (общий вход для нескольких устройств/телефона, авторизация по паролю, стабильная «прихожая»); хочешь скрыть домашний IP от residential-провайдера; либо шлюз из РФ напрямую не открывается (релей дотянется до него «из-за границы») |

> Оба способа дают **резидентный выходной IP** (не датацентровый) — именно это обходит фильтры, режущие серверные ASN (Antigravity и агрессивный анти-бот). Бери **static/ISP** (фиксированный IP), не rotating — иначе ломаются логины. В способе B зарубежный Dante-VPS становится лишним: residential и есть «заграница».

## Получить Antigravity Pro: Jio (бесплатно) vs Plati/G2A (платно)

> Если у тебя есть индийская Jio-SIM — **делай Jio**. Платные «18-month Global» лоты на [G2A](https://www.g2a.com/) с высокой вероятностью = перепроданные те же Jio-промокоды. Платить за то, что можно получить даром.

### Путь 1 — Jio (бесплатно, 18 месяцев)

**Условия:** активный тариф Jio Unlimited 5G от **₹349** (prepaid/postpaid), 18+.

**Шаги:**
1. SIM в телефоне, тариф ₹349+ активен.
2. Открой **MyJio** → вход по Jio-номеру.
3. Баннер **Google Gemini Offer** на главной → тапни.
4. Войди своим **Gmail** → подтверди. Google AI Pro привязывается к аккаунту на 18 мес.
5. Заходи в Antigravity этим Google-аккаунтом (через туннель, если режет по сети).

**Два затыка из России — проверь заранее:**
- **OTP на индийский номер.** Вход в MyJio требует SMS-код. Из РФ придет, только если на номере **активен international roaming** (prepaid — нужен IR-пакет; postpaid — включить роуминг). Нет приема SMS — стоп.
- **Поддержание тарифа.** ₹349+ держать активным (пополнение из-за рубежа — индийской картой/средствами). Часть офферов требует, чтобы квалифицирующий тариф оставался активным — см. [FAQ оффера](https://www.jio.com/help/faq/mobile/offers/google-gemini-offer/).

**Известный глюк:** «в MyJio активно, в Google-аккаунте не видно» → регион Google-аккаунта = **Индия**, повторный вход, время на синхронизацию ([тред в саппорте](https://support.google.com/gemini/thread/401425383)).

После активации подписка живет **на Google-аккаунте** и работает глобально; SIM нужен в момент активации (и, возможно, для поддержания тарифа).

### Путь 2 — Plati / G2A (платно, запасной)

Выбор типа лота критичен:

| Тип лота | Брать? | Почему |
|---|---|---|
| **«На вашу почту» / activation link** (вешают на ТВОЙ Gmail через `payments.google.com/redeem`) | ✅ да | аккаунт остается твой |
| **«Готовый аккаунт»** (чужой логин/пароль) | ❌ нет | не твой; могут отозвать / сменить пароль / забанить |

**Активация (тип «на вашу почту»):** [payments.google.com/redeem](https://payments.google.com/redeem) → вход своим Google-аккаунтом → ввод кода → подписка добавлена. Либо продавец добавляет email по ссылке.

**Цены (июнь 2026):** [Plati](https://plati.market/games/gemini/1541/) — Google AI Pro 6 мес. ≈ **1290₽** (промо от ~600–660₽); [G2A](https://www.g2a.com/) — 1/3/6/12/18 мес.

**Правила, чтобы не потерять деньги/аккаунт:**
- После активации **не меняй регион и платежный метод** — [G2A](https://www.g2a.com/) прямо предупреждает: это блокирует аккаунт.
- Нельзя «сложить» поверх активной подписки — гаси код после окончания текущей.
- Продавец с высоким рейтингом и **гарантией/заменой**; возвратов нет.
- Активируй на **отдельном** Google-аккаунте, не на основном — промо-апгрейды Google может отозвать.

### Связка с туннелем

Подписка решает **вход/квоту**, но **сетевой доступ** к Antigravity всё равно может требовать туннель (US-IP). Подписка ≠ доступ из РФ — нужно и то, и другое.

### ⚠️ Нюанс региона: аккаунт Индия ↔ туннель США

Здесь единственное место, где две части могут конфликтовать:
- Jio-оффер ориентирован на регион **Индия**, и для синхронизации Pro регион Google-аккаунта обычно должен быть **India**.
- Antigravity же ты дергаешь через **US-туннель**, и он может капризничать по региону.

Сама подписка привязана к **Google-аккаунту и от сети не зависит** — то есть Pro, выданный в индийском регионе, остается на аккаунте при заходе через US-IP. Но если Antigravity упрется в регион, развилка такая:

| Что показывает | Что делать |
|---|---|
| Pro активен, Antigravity пускает через US-туннель | ничего, всё сошлось |
| Pro активен, но Antigravity ругается на регион | попробовать туннель **India** для Antigravity, или согласовать регион аккаунта с регионом туннеля |
| MyJio «активно», но в Google-аккаунте Pro не видно | регион аккаунта = **India**, повторный вход, выждать синхронизацию (см. глюк выше) |

> 📌 Точную связку (какой регион аккаунта + какой регион туннеля) подберем по факту — на этапе активации зафиксируй, **что именно показывает MyJio и страница подписки Google-аккаунта**, и от этого пляшем.

## Telegram через прокси (без VPN, VPS и Proxifier)

Для Telegram вся эта цепочка не нужна: у него **встроенная поддержка прокси**, а собственный протокол **MTProto** специально сделан под обход блокировок — маскирует трафик под обычный HTTPS, DPI его обычно не отличает. Ни VPN, ни VPS, ни Proxifier.

> Контекст: с **февраля 2026** РКН официально *замедляет* Telegram (режет скорость медиа — голосовых, видео, картинок). MTProto-прокси — штатное лекарство.

### Как включить (без своего сервера)

1. Возьми готовый **MTProto**-прокси из публичного источника:
   - канал [@ProxyMTProto](https://t.me/s/ProxyMTProto) и подобные — публикуют свежие; тап по кнопке/ссылке сразу добавляет;
   - боты-раздатчики прокси прямо в Telegram;
   - авто-обновляемые списки на GitHub (напр. [Grim1313/mtproto-for-telegram](https://github.com/Grim1313/mtproto-for-telegram), обновление раз в 12 ч).
2. Либо вручную: **Настройки → Данные и память → Прокси → Добавить прокси → MTProto** → впиши `Server` / `Port` / `Secret`.
3. Либо просто тапни ссылку вида `tg://proxy?server=...&port=...&secret=...` — Telegram сам предложит подключиться.

Тип бери **MTProto**, а не SOCKS5: он шифрует и маскирует трафик. SOCKS5 тоже поддерживается, но к блокировке устойчив хуже.

### Подводные камни (честно)

- **Бесплатные публичные прокси нестабильны:** «умирают» от нагрузки, а РКН блокирует их целыми подсетями (1 апреля 2026 разом отвалились тысячи). Держи 2–3 запасных и обновляй из канала.
- **Реклама:** владелец бесплатного прокси обычно закрепляет **спонсорский канал** вверху списка чатов — это плата за «бесплатно». Его можно скрыть/отписаться.
- **Доверие:** оператор прокси твою переписку **не прочитает** (Telegram шифрует клиент-сервер сам), но видит метаданные соединения. Для мессенджера приемлемо; для чувствительного — лучше свой прокси (ниже).

### Если хочешь стабильнее и приватнее (опционально, уже с сервером)

- **Нацелить Telegram на твой РФ-релей.** У тебя уже поднят [scripts/setup-ru-relay.sh](scripts/setup-ru-relay.sh): в Telegram добавь прокси **SOCKS5** → `IP_РФ_сервера:1080`, логин `relayuser` + пароль. Личный канал, без Proxifier.
- **Свой MTProto-прокси с Fake TLS** на зарубежном VPS — топ по надежности и маскировке (прикидывается HTTPS реального сайта, РКН не ловит по кластерам). Гайды на Habr: [MTProto + Fake TLS за 5 минут](https://habr.com/ru/articles/1010942/), [«партизанский» прокси под онлайн-магазин](https://habr.com/ru/articles/995102/), [способы обхода замедления Telegram](https://habr.com/ru/articles/1014934/). Готовый скрипт уже есть — [scripts/setup-foreign-mtproxy.sh](scripts/setup-foreign-mtproxy.sh): ставит mtg, генерит Fake-TLS-секрет (маскировка под домен) и выдает готовую `tg://proxy`-ссылку (подробности — [scripts/README.md](scripts/README.md)).

## Если не работает

| Симптом | Причина / решение |
|---|---|
| `systemctl status danted` → **failed** | `journalctl -u danted -n 30 --no-pager`. Чаще всего — неверное имя интерфейса в `external:` (шаг 4) или опечатка в конфиге |
| Proxifier **Check** не проходит | порт закрыт — проверь ufw (шаг 7) **и** облачный фаервол провайдера |
| Прокси работает, но Claude/Antigravity всё равно блокируют | IP попал в черный список → пересоздай VPS с другим IP/в другом городе США. Для Antigravity — еще проверь страну Google-аккаунта (см. 0.2) |
| `curl` показывает не тот IP / ошибку авторизации | проверь пароль `proxyuser` (шаг 5) и `socksmethod: username` в `socks pass` |
| Antigravity видит дата-центровый IP и режет | перейди на резидентный прокси — [Вариант Б](#вариант-б--резидентный-прокси-без-своего-сервера) (бери static/ISP, US) |
| Резидентный прокси: логин слетает посреди сессии | rotating-IP сменился — включи **sticky session** и скопируй username из генератора целиком; либо возьми **static (ISP)** |
